# 요구 명세서 (PRD)

## 프로젝트명

반도체 시료 생산주문관리 시스템
(Sample Order Management System)

---

# 1. 프로젝트 개요

## 목적

반도체 시료(Sample)의 등록, 주문, 생산, 재고, 출고 과정을 통합 관리하는 콘솔 기반 시스템을 개발한다.

기존의 엑셀 및 수기 관리 방식에서 발생하는 주문 처리 오류, 재고 관리 문제, 생산 현황 파악의 어려움을 해결하는 것을 목표로 한다.

---

# 2. 개발 목표

* 콘솔 기반 애플리케이션 구현
* MVC 아키텍처 적용
* 데이터 영속성 지원
* 단위 테스트 작성
* Agentic Engineering 기반 개발
* Clean Code 원칙 준수
* 유지보수 가능한 구조 설계

---

# 3. 핵심 도메인

## Sample (시료)

### 속성

* Sample ID
* Name
* Average Production Time
* Yield Rate
* Stock Quantity

### 규칙

* Sample ID는 유일해야 한다.
* 수율은 0 ~ 1 사이 값이어야 한다.
* 등록된 시료만 주문 가능하다.

---

## Order (주문)

### 속성

* Order ID
* Sample ID
* Customer Name
* Quantity
* Status
* Created At

### 주문 상태

| 상태        | 설명    |
| --------- | ----- |
| RESERVED  | 주문 접수 |
| PRODUCING | 생산 진행 |
| CONFIRMED | 출고 대기 |
| RELEASE   | 출고 완료 |
| REJECTED  | 주문 거절 |

---

## Production Job (생산 작업)

### 속성

* Production ID
* Order ID
* Sample ID
* Required Quantity
* Actual Production Quantity
* Total Production Time

### 규칙

실 생산량 계산

ActualProductionQuantity

= ceil(부족수량 / (수율 × 0.9))

총 생산 시간 계산

TotalProductionTime

= 평균생산시간 × 실생산량

---

# 4. 기능 요구사항

---

## FR-01 시료 등록

### 설명

신규 시료를 등록한다.

### 입력

* Sample ID
* Name
* Average Production Time
* Yield Rate

### 검증

* ID 중복 불가
* 이름 공백 불가
* 생산시간 0 초과
* 수율 0 초과 1 이하

### 결과

시료가 저장된다.

---

## FR-02 시료 조회

### 설명

전체 시료 목록을 조회한다.

### 출력

* Sample ID
* Name
* Yield Rate
* Average Production Time
* Current Stock

---

## FR-03 시료 검색

### 설명

시료 이름으로 검색한다.

### 입력

* Sample Name

### 결과

일치하는 시료 목록 출력

---

## FR-04 주문 접수

### 설명

고객 주문을 등록한다.

### 입력

* Sample ID
* Customer Name
* Quantity

### 결과

상태를 RESERVED로 생성한다.

---

## FR-05 주문 승인

### 설명

RESERVED 주문을 승인한다.

### 처리 규칙

재고 >= 주문수량

→ 상태 변경

RESERVED → CONFIRMED

재고 < 주문수량

→ 생산 큐 등록 ( 부족한 수량 만큼 )

→ 상태 변경

RESERVED → PRODUCING

---

## FR-06 주문 거절

### 설명

RESERVED 주문을 거절한다.

### 결과

상태를 REJECTED로 변경한다.

---

## FR-07 생산 큐 관리

### 설명

생산 대기 작업을 FIFO 방식으로 관리한다.

### 규칙

* 먼저 등록된 작업 우선 생산
* 생산 큐 조회 가능

---

## FR-08 생산 처리

### 설명

생산 작업을 수행한다.

### 처리 과정

1. 생산 큐에서 작업 선택
2. 부족 수량 계산
3. 실 생산량 계산
4. 재고 증가
5. 주문 상태 변경

### 상태 변경

PRODUCING → CONFIRMED

---

## FR-09 출고 처리

### 설명

CONFIRMED 주문을 출고한다.

### 처리

* 재고 차감
* 상태 변경

CONFIRMED → RELEASE

---

## FR-10 주문 모니터링

### 설명

상태별 주문 현황을 조회한다.

### 출력

* RESERVED 개수
* PRODUCING 개수
* CONFIRMED 개수
* RELEASE 개수

※ REJECTED 제외

---

## FR-11 재고 모니터링

### 설명

시료별 재고 상태를 조회한다.

* 주문대비 재고 수량에 따라 상태도 표기

  * 요청 재고를 확인하고 주문수량 대비 상태 확인 
  * 주문 가능 재고: confirmed , producing  수량을 고려해서 해당 주문에 대한 수량은 재고 에서 제외

### 상태 규칙

| 상태 | 조건            |
| -- | ------------- |
| 여유 | 주문대비 재고 충분    |
| 부족 | 주문 대비 재고수량 부족 |
| 고갈 | 재고 0          |

# 추가 요구사항 및 설계 제약사항

기존 PRD 외에 아래 규칙을 반드시 준수하여 설계 및 구현한다.

---

# 1. 주문과 시료는 독립적으로 관리한다

Sample은 제품 마스터 정보만 관리한다.

```text
Sample
- sampleId
- name
- averageProductionTime
- yieldRate
```

Order는 고객 주문만 관리한다.

```text
Order
- orderId
- sampleId
- customerName
- quantity
- status
```

Inventory는 재고만 관리한다.

```text
Inventory
- sampleId
- totalStock
- allocatedStock
```

주문(Order)은 재고를 직접 소유하거나 변경하지 않는다.

---

# 2. 생산 여부는 주문 승인 시점에 단 한번 결정한다

주문 승인 시점에 주문 가능 재고를 확인한다.

```text
availableStock = totalStock - allocatedStock
```

주문 승인 시

```text
availableStock >= orderQuantity
```

이면

```text
RESERVED -> CONFIRMED
```

으로 변경한다.

이 경우 주문 수량만큼 allocatedStock을 증가시킨다.

---

주문 승인 시

```text
availableStock < orderQuantity
```

이면

```text
RESERVED -> PRODUCING
```

으로 변경한다.

ProductionJob을 생성한다.

---

중요

생산 필요 여부는 주문 승인 시점에 단 한번만 판단한다.

생산 작업이 생성된 이후에는 재고 상황이 변경되더라도 생산 여부를 재판단하지 않는다.

생산 작업은 취소되지 않는다.

---

# 3. 생산라인은 1개만 존재한다

생산라인은 동시에 하나의 작업만 수행한다.

생산 작업은 FIFO 방식으로 처리한다.

```text
Queue<ProductionJob>
```

구조를 사용한다.

---

# 4. 생산 작업 생성 시 모든 계산을 완료한다

ProductionJob 생성 시 아래 정보를 모두 계산하여 저장한다.

```text
requiredQuantity
actualProductionQuantity
totalProductionTime
startTime
expectedFinishTime
```

생산 시작 후 부족수량을 다시 계산하지 않는다.

---

# 5. 생산량 계산 규칙

```text
actualProductionQuantity
=
ceil(
requiredQuantity
/
(yieldRate * 0.9)
)
```

예시

```text
requiredQuantity = 100
yieldRate = 0.1

actualProductionQuantity
=
ceil(100 / (0.1 * 0.9))
=
1112
```

---

# 6. 생산 진행 중 발생한 초과 생산분은 즉시 사용 가능하다

생산 과정에서 생성된 정상 시료는 실시간으로 재고에 반영한다.

예시

```text
주문1 수량 = 100
실생산량 = 120
```

현재 생산 완료 수량

```text
115
```

이라면

```text
주문1 확보 수량 = 100
가용 수량 = 15
```

으로 간주한다.

신규 주문이 15개 들어오면

```text
CONFIRMED
```

상태로 처리 가능하다.

---

# 7. 재고 정책

현재 재고

```text
totalStock
```

실제로 생산 완료된 정상 시료 수량

---

예약 재고

```text
allocatedStock
```

이미 주문이 확보한 수량

---

주문 가능 재고

```text
availableStock
=
totalStock - allocatedStock
```

신규 주문 승인 여부는 반드시 availableStock 기준으로 판단한다.

---

# 8. 생산 완료 처리

생산이 완료되면

```text
PRODUCING -> CONFIRMED
```

로 상태를 변경한다.

생산 작업은 생산 큐에서 제거한다.

---

# 9. 시간 처리 설계

운영 환경에서는 실제 시간을 사용한다.

```cpp
SystemTimeProvider
```

를 사용한다.

생산 완료 여부는 실제 현재 시간 기준으로 계산한다.

---

# 10. 테스트 가능성 확보

시간 의존 로직은 반드시 DI를 적용한다.

모든 서비스는 시간 조회 시 직접

```cpp
std::chrono::system_clock::now()
```

를 호출하지 않는다.

반드시

```cpp
ITimeProvider
```

를 통해 현재 시간을 조회한다.

---

Unit Test에서는

```cpp
MockTimeProvider
```

또는

```cpp
FakeTimeProvider
```

를 주입하여 테스트한다.

예시

```cpp
mockTime.Advance(std::chrono::seconds(300));
```

처럼 시간 경과를 시뮬레이션할 수 있어야 한다.

실제 300초를 기다리는 테스트는 허용하지 않는다.
